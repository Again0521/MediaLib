# MediaLIB 开发说明

2026-06-05 补充：除音乐展开页外，普通页面、弹窗、工具条、设置、菜单和重复控件必须先对照 `MediaLIB_设计系统标准.md`。页面使用 `PageHeader`，普通编辑/添加/重命名/确认类弹窗使用 `AppSheetHeader`、`AppInfoNote`、`AppSheetActionFooter` 和 `appSheetChrome`，页内工具条使用 `AppSurfaceToolbar`；工作台型大弹窗可保留页面级结构，但仍要遵守反馈、文案和静态玻璃性能约束。会打开后续流程的按钮使用省略号，立即动作不使用省略号。所有可点击控件必须有明确 hover 或点按反馈。`LiquidGlassButtonStyle`、`RepeatedGlassButtonStyle`、`HeaderActionGlassButtonStyle`、`GlassMenuButton` 和 `GlassCapsuleControl` 都承担这条基础交互约束；批量操作栏和返回顶部按钮不能退回裸 `.plain` 文字/图标按钮。反馈只使用边缘光、描边、亮度、透明度或轻量色彩变化，不使用 scale 或阴影 y 偏移造成布局晃动。健康结果必须绑定仍存在且开启 `includeInHealthCheck` 的来源，删除来源或关闭检查要立即失效旧缓存；保险库播放记录仅在解锁后进入“正在观看/已观看”并支持清除。选择菜单使用当前文字自适应宽度；媒体源行是例外，分类、服务器库、健康/元数据参与、写入源目录偏好和痕迹同步都必须放进每行设置弹窗，不要恢复行内选项框。智能歌单新建入口位于“音乐 > 歌单”，设置页按视频播放、音乐播放、通用控制和元数据子类分组。主窗口透明标题栏逻辑必须排除 `NSPanel`，避免系统文件选择/保存框透明异常；强调按钮使用深蓝单层薄玻璃，侧栏内联新建入口与普通行严格对齐。派生缓存保持单次全库分桶，“正在观看”读取预构建公共/保险库缓存，喜欢/想看只更新目标父级子项；单歌单和外部播放器发现也必须保持直接查询/缓存路径。

## 技术栈

- UI：SwiftUI，必要时使用 AppKit。
- 代码健康：模型层属性必须保持纯内存计算，不得在 `MediaItem` / `MediaSource` 这类会进入列表、卡片或筛选流程的类型里同步调用 `FileManager`。本地文件可达性、来源离线状态和缓存有效性统一走 `AppState` 派生缓存、后台任务或明确的服务方法；不要恢复已删除的 `MediaItem.isPlayable` / `MediaSource.exists`。锁定保险库相关任务、日志、健康中心、确认弹窗和重新挂载提示必须使用安全标签：用户可见名称按解锁状态泛化，健康行的路径详情要在界面层兜底隐藏，日志始终不得带出保险库条目名、来源名、路径、服务端地址或底层错误对象。
- 设置与配色：设置页“关于软件”入口使用普通设置分组与标准 `appSheetChrome` 弹窗，展示项目地址、作者和联系方式，不引入独立窗口或自定义系统面板；关于页左侧图形应引用 `AppIcon.png` 等真实软件 Logo，信息行使用固定图标列、标题列和值列保证垂直重心和文字起点对齐。普通页面配色仍由 `AppThemePreset` / `AppThemeTokens` / `AppColors` 三层负责，2026-06-07 后 hover / pointer light、`PlayfulSymbolIcon`、默认音乐封面和播放器外部氛围色都必须从当前主题左上光线与高亮色共同派生，不能回退固定系统蓝；无专辑色的音乐底栏/队列状态也优先回退 `AppColors.selectedGlassTint` 或 `pointerLightTint`。
- 2026-06-07 返回位置约束：视频海报墙、首页海报型标签、首页总览横向看板、全局搜索结果和片库健康中心都通过 `AppState.selectedItemReturnAnchorID` 记录打开详情前的来源锚点，返回后由对应 `ScrollViewReader` / `PosterGridList` 用禁用动画的 `Transaction` 静默定位到原卡片或原行，再清空锚点。新增详情入口如果会替换主 detail 区，应在设置 `selectedItem` 前写入来源条目 ID，并给原列表行 `.id(item.id)`；切换侧栏 destination 时必须清空该锚点，避免跨页面误滚动。`DetailView` 页头显示当前二级目录标题，作品名留在 hero 内容区，不要在页头重复作品名。
- 视觉/动画：音乐展开页的大背景必须走 `MetalAlbumBackdropView` 的不透明 Metal 底板，shader 最终输出 alpha=1，不能使用整窗 `NSVisualEffectView(.behindWindow/.underWindow)`、SwiftUI 全屏 material 或可透出桌面的背景。歌词卡片、控制栏、弹层、收起按钮和底部迷你播放器玻璃底座可使用局部 `NSVisualEffectView(.hudWindow, .withinWindow)` 或伪玻璃预渲染层；不要把 SwiftUI 原生 `.regularMaterial` / `.thinMaterial` / `.ultraThinMaterial` 铺在动态大面积热路径上。歌词卡片、控制栏和收起按钮保留低白度填充、静态雾面纹理和中性边缘光，不叠专辑色 tint、彩色描边或彩色浸染；歌词大卡片不要再用整块 live material 或连续 pointer 采样；它的上下景深应由连续渐变和描边完成，中心舞台光使用 `CAGradientLayer`，面板/控制栏阴影使用带 `shadowPath` 的 `CALayer`，避免出现横向材质断层和大面积 SwiftUI blur/shadow 热路径。展开页底层封面氛围使用低频封面颜料场与专辑三色语义校色，不要恢复 SwiftUI 整屏实时高斯 blur。不要把 material、blur、大阴影或连续 pointer 采样重新铺回列表行、海报卡、设置分组和普通按钮。展开/收起动画要继续通过 `AppMotion` 分帧处理窗口 chrome、overlay、侧栏和静态专辑光层：外层播放器 overlay 只能做整窗 opacity，不做 scale，展开时先挂载 overlay 再下一帧用 AppKit 隐藏标题/侧栏切换按钮，并保持 `NSToolbar.isVisible` 与 toolbar safe area 稳定；顶部白条只能通过透明化/清空 AppKit chrome 背景层让同一个 `MusicPlayerView` 背景露出解决，不要新增单独顶栏背景或材质层；收起时先回到底栏再恢复 chrome；非全屏窗口转场期间由 `MainWindowToolbarVisibilityGuard` 保持窗口尺寸；展开态必须隐藏侧栏展开按钮并保持红黄绿按钮可见，不要通过隐藏整条工具栏来处理顶部视觉。
- 2026-06-09 音乐展开页最新视觉边界：L0 背景继续只由 `MetalAlbumBackdropView` 的单个 `MTKView` 绘制不透明专辑色板、低频封面颜料场、三色 mesh、整屏玻璃高光、轻折射波纹和 grain；封面输入约 160px，`paintPalette` 降到约 38px 网格后 15px 高斯柔化并做轻对比清洁，去掉可辨识文字/人物结构。切歌时可提前加载新封面纹理，但 shader 是否使用封面纹理由 `artworkOpacity` 和纹理 alpha 共同控制，不要在 ready 前清空到灰底或让透明 fallback 压黑。歌词卡、控制栏、收起按钮统一通过 `FloatingLyricsGlass` + `LyricsCardEffectLayerView` 的 CA 层绘制常态玻璃底、上下雾区、内阴影和 pointer 扫光；新增展开页玻璃组件应复用这条路径。`Sources/MediaLib/Views` 下只保留这一份效果层源码，不要把带空格、编号或“备份”含义的 `.swift` 副本留在 `Sources` 目录内，因为 SwiftPM 会同样编译它们并造成同名类型重复定义。三块玻璃整体保持更透明的 `.hudWindow` / `.withinWindow` 材质，白色 frost、纹理和顶部 specular 都必须克制；静态边缘光、pointer 光和舞台光使用专辑 glow 的低饱和 tint，但玻璃底材质不能改成专辑色实底。只有真正的歌词卡片允许启用 `centerClarity`，中心通透感由静态 fill 渐变与 `LyricsCardEffectLayerView` 承载，不再给 material 或磨砂纹理挂 SwiftUI 渐变 mask，控制栏与收起按钮不得启用中心透明通道。封面外圈发光由 `AlbumBlurredCoverGlowLayer` 使用 `AlbumCoverGlowBakeCache` 后台预烘焙 artwork 原图得到的 far ambient / mid spill / near bloom 三层透明 image-based glow、`AlbumProjectedCoverGlowLayer` 的页面底层同心环境光池，以及方向化 `AlbumLightSpillOverlay` 的组件受光共同表达；投射光只复用预烘焙封面复制图，并通过同心方形 frame 与径向羽化控制影响范围，不得改回运行时大半径 SwiftUI blur。预烘焙必须先校正 Core Image Y 轴，再按真实可见封面裁切并用邻近边缘像素向外延展；最终 alpha 必须乘以亮度门控，近黑/黑场只提供遮光和深度，不得外扩成灰黑脏光，并通过 blur、alpha mask、径向尾部衰减、opacity clamp 和 screen / plusLighter 合成控制亮度。控制栏高度固定为 `MusicPlayerVisualTokens.Controls.expandedHeight`，并由外层固定 frame 承担尺寸约束；内部不再保留独立高斯取色底，受光只通过 `AlbumLightSpillOverlay` 进入，状态行或 material 不得参与高度计算。封面本体描边和 pointer 边缘光以中性玻璃高光为主，palette 色只保留极低强度，避免贴边层被误认为封面 glow 偏色。底板取色必须结合两条路径：彩色封面先用低频封面颜料场保留空间色彩关系，再把 primary/secondary/accent 清洁化为受控明度/饱和度的 tonal 角色色后参与语义调色；低彩/白灰封面用真实中性平均保留白/灰/黑主调，只有彩色面积占比足够明确时才允许极弱色相提示，不得强制提饱和成粉色。不要恢复单色阴影、预烤单张 glow、全屏 SwiftUI glow、运行时三层 SwiftUI blur、整图全局 blur 后外扩的 glow、轻纹理单色底板或第二套冲突光晕。定时歌词浏览态必须完全取消行 blur，停止浏览后保留约 4 秒清晰阅读窗口，再按 `lyricBrowseBlurProgress` 渐进恢复。
- 2026-06-09 晚间修订：音乐展开页以效果优先但继续保留 Metal/预烘焙性能路径。L0 `MetalAlbumBackdropView` 必须在 shader 中采样 `paintPalette` 的封面低频高斯色场作为主色源，当前约 38px 网格、15px 柔化，并由 `glassifyAlbumColor` 清洁成透亮色温场；不要再退回只用 primary/secondary/accent 拼 mesh 的单色底板。L1 是整窗厚玻璃盖板，只允许轻白雾、折射波纹、边缘高光和高光压缩来表达玻璃，不要铺全屏 SwiftUI material 或实时 blur。封面发光继续走 `AlbumBlurredCoverGlowLayer` -> `AlbumCoverGlowBakeCache`，近黑/黑色通过亮度门控降权，不能引入与封面亮部无关的单色光。`AlbumProjectedCoverGlowLayer` 与封面中心同心扩散，reach 同时参考歌词卡左缘和控制栏上边界；不要再用横向拉伸、偏移和多重线性 mask 把光做成下沿色带。歌词卡、控制栏、收起按钮本身保持中性透明磨砂玻璃，专辑色只通过方向化 `AlbumLightSpillOverlay`、静态边缘 tint、pointer tint 和中心舞台光作为外部受光层进入：歌词卡从左缘受光、控制栏从上缘受光、收起按钮从左上受光，且这些受光层必须位于玻璃背景之上、文字按钮之下。
- 2026-06-09 夜间补充：音乐展开页底板和封面光按“黑色不发光、玻璃受彩色光”规则收口。专辑 palette 采样会降低近黑像素权重；`AlbumCoverGlowBakeCache` 的最终 alpha 会再乘以低频色场亮度门控，黑底、暗边和近黑背景只保留空间深度，不再向外扩散成灰黑脏光。`MusicPlayerMetalBackdropView` 的 `paintPalette` 只做极轻 vibrance，`glassifyAlbumColor` 不再把暗部硬抬成亮色；shader 会在封面低频颜料场上叠加小幅色相和声场，让底板和封面同源但不完全同向。白色 veil、sweep、边缘空气光和歌词卡中心舞台光都必须以专辑 `glowPrimary/glowSecondary` 为主，白色只保留薄镜面 specular；不要把玻璃重新照成大面积白雾。高饱和封面要通过 role color 清洁、亮区降彩和最终 clamp 控制信息量，不能让底板比封面更艳或更刺眼。
- 2026-06-09 歌词与玻璃微调补充：`LyricSourceParser.coalescedTimestampLines` 不能再把 80ms 内同时间戳歌词无条件合并；检测到含假名日文原文与纯汉字中文翻译时必须保留为独立相邻行并维持文件显示顺序，播放/滚动锚点优先落在含假名日文原文，翻译行只作为同时间戳伴随行显示。歌词卡中心舞台光必须继续位于文字层下方，使用宽椭圆 + 低峰值专辑色径向光，不用高 alpha 白光。若歌词卡、控制栏或收起按钮再次发灰，优先降低 `MusicGlassSurfaceRole.materialOpacity`、内部黑色 shade 和中性 tint，再用 `LyricsCardEffectLayerView` 的上下雾面/发丝高光补厚度；不要新增全屏 blur 或把玻璃底改成专辑色实底。底板只允许小幅提高 `paintPalette` saturation/vibrance、role color 饱和上限和 `kChromaBoost`，仍要保持黑色不发光、亮区不刺眼。
- 音乐展开页在全屏覆盖稳定后会把背后的 `NavigationSplitView` 从视图树卸载，释放列表、海报墙和筛选结果的资源；收起前先无动画把下层界面预热挂回，再让播放器退场，用户只会看到平滑回落到底栏。展开页全屏 Metal 底板、低频颜料纹理和近场专辑光应首帧挂载，不要再用 520ms/730ms 延迟插入图层，否则会重新产生展开后断层和稳定态合成峰值。歌词卡片、控制栏和收起按钮继续共用 `FloatingLyricsGlass`，但歌词大卡只走静态雾面、专辑色高光和 CA 光影，控制栏/弹层可保留小面积 live glass，不能把该路径扩回大面板。
- 页头与控件层级：带搜索、扫描、清除记录等操作的页面必须复用 `PageHeader` 的“标题左侧 + 右侧操作区”布局，搜索框、扫描和清除记录按钮位于标题栏右侧，操作区下边界与标题栏下边界对齐；这些页头控件必须共用 `HeaderControlGlassBackground` 的暖白系统玻璃、左上浅米白/香槟高光和轻接触阴影，不要重新叠冷蓝填充或大投影。`GlassSearchField` 宽度按占位或输入文字在 `minWidth/maxWidth` 之间弹性计算，页面应为页头搜索框设置合理最大宽度，避免标题被挤压；设置页、智能集合和智能歌单的菜单控件使用当前选项文字自适应宽度并预留箭头空间，不要恢复固定大宽度造成右侧留白。普通页面背景、卡片、搜索框、菜单和按钮要保持暖白半透明玻璃、香槟左上受光、右下轻暗边和低噪声层级，冷蓝只保留给应用图标、少量强调按钮和轻微折射，避免背景、卡片、按钮再次全变成正白或泛蓝。
- 右键菜单：同一个右键菜单内只要有一个条目带图标，其余可操作条目也必须使用 `Label` 或等价图标列补齐左侧图标，避免文字起点不齐。视频缓存、重分类、播放/外部打开、收藏/想看/已观看、音乐队列、添加到歌单、歌单管理等上下文菜单都应保持图标覆盖；媒体类型图标统一走 `MediaType.systemImage`，不要在各菜单里重复硬编码。视频播放器画面右键使用 `PlayerContextMenuOverlay` 自绘玻璃菜单，而不是系统 `contextMenu`，以便继承播放器内玻璃和控制层配色；播放器右键菜单、齿轮设置、轨道/字幕/画质/章节弹层、进度预览气泡、音量 HUD、跳过片头/片尾浮层、加载/错误覆盖层和截图提示都必须接入 `VideoControlPalette`，不要再硬编码白色文字、图标或选中态。播放器玻璃填充保持均匀的上下轻高光，斜向效果只可用于细边框折射，不能把控制条底色做成明显从左到右变透明。菜单和弹层只允许通过 `PlayerControllerProjection` 观察所需字段；不要让右键菜单、音轨/字幕/倍速/音量弹层或 `MpvPlayerView` 订阅完整 `MpvPlayerController` 的高频进度变化，也不得恢复全局鼠标监听或把菜单状态接到播放进度热路径。
- 卡片交互：视频海报卡、音乐专辑卡和歌单卡的 hover 美化应优先使用固定高光、封面描边、轻微 scale 和 `repeatedSurfaceHover`，并读取 `suppressPointerHoverDuringScroll` 与 `accessibilityReduceMotion`。不要为了“更有动画”把这些滚动热路径改回实时 material、大投影或连续鼠标光。
- 纠偏约束：普通卡片 hover 不允许再对整卡做 scale/offset，避免按钮点击时底层卡片一起移动；需要层级时使用边缘、描边、接触阴影和局部高光。所有按钮、菜单和输入框的绘制层要裁剪到自身圆角，避免底部或边缘露出直角色块。`FloatingLyricsGlass` 的装饰层必须 `allowsHitTesting(false)`，且歌词面板不要再使用固定圆心的径向高光，避免左上角出现与专辑无关的圆形印记。
- 按钮交互：普通按钮、页头按钮、重复列表按钮和音乐图标按钮不要用按压 scale 或阴影 y 偏移表达点击，否则会造成卡片/按钮条跳动。按压反馈优先用透明度、边缘光或描边变化；页头布局不要随按钮状态隐式动画。
- 播放：视频使用 `PlayerView.swift` 的 SwiftUI 独立窗口，通过 `LibMpvClient.swift` 动态加载随 App 分发的 `libmpv` 核心，并用 libmpv render API 渲染到应用内 `NSOpenGLView`；音乐使用原生 AVFoundation 音频后端。视频和音乐播放器设置分离，AVFoundation 也用于音频标签读取和封面/预览帧抽取；播放器内“保存当前画面”走 mpv `screenshot-to-file`，输出到用户图片目录下的 `MediaLIB 截图`，不得写回或修改源视频文件。`AirPlayRoutePickerButton` / `AirPlayRoutePickerSession` 封装 AVKit 路由选择器并在 `MpvPlayerController` 中持久绑定最近的 `AVPlayer`；每个可见 AirPlay 控制使用 SwiftUI 按钮接管点击，再主动触发隐藏的原生 route picker，避免依赖透明 NSView 覆盖命中导致展开页或底栏不可点击。音乐 AirPlay 不再提供本机同播路径，主音乐 `AVPlayer` 始终允许系统外部播放；保留 `AppSettings.keepLocalAudioWithAirPlay` 仅用于旧设置解码兼容，不再驱动 UI 或播放器行为。视频路线选择开始和结束时通过 AVPlayer 代理尝试同步 AirPlay 外部播放，路线选择结束后要延长外部播放状态探测；macOS 的 `AVRoutePickerView.prioritizesVideoDevices` 不可用，视频入口只能通过绑定视频代理播放器、视频图标和后续探测来保证行为。`SystemMediaIntegration` 负责 macOS 菜单栏/系统媒体键的基础播放命令和 Now Playing 信息。
- 音乐封面：扫描时只提取单曲音频内嵌 artwork；不要把同目录 `cover/folder/poster` 作为音乐封面兜底，联网元数据不能覆盖内嵌封面。
- 音乐标签编辑：`MusicTagEditingService` 提供精简原生写回核心，`MusicTagScraperSheet` 提供设置页“音乐元数据获取”工作台。当前没有 vendoring GPL/额外限制的 `music-tag-web`，也没有打包 Python/Node 常驻服务；只参考 MIT `music-tag` 的统一标签抽象思路。默认只更新 `MediaRepository` 中的索引，用户显式打开“写入文件”时才通过 ffmpeg 复制音频流并写入标签，生成临时文件并校验后才替换原文件。
- 封面：本地图片优先，缺失时可用 AVAssetImageGenerator 抽帧或 AppKit 绘制默认封面。
- 保险库：Application Support 的 `MediaLib/Credentials` 文件保存 PIN 哈希，LocalAuthentication 调用 Touch ID / 设备认证；不要读写或删除系统 Keychain。

字段、`prefer_metadata_write_to_source`、`remote_trace_sync_mode`、`user_rating`、手动集合表、`video_smart_collections.rules_json`、手动/智能集合 `show_on_home`、Emby/Jellyfin/Plex 来源 `selected_emby_library_ids` 和 `video_offline_subscriptions` 基础上，新增 `metadata_correction_history`、`sync_conflicts`、`remote_connector_accounts`、`local_user_profiles`、`profile_media_state`。该字段名沿用 Emby 以兼容旧库，保存 Emby/Jellyfin/Plex 服务器库 ID 列表，空值表示兼容旧行为的全库同步，非空时仅同步指定服务器库并通过 `replaceRemoteItems` 清理未选库的内部索引。自动缓存订阅保存系列 ID、标题、模式、集数窗口、目标季、清晰度、启用状态、暂停到期、计划到期和网络策略；未看窗口可通过右键菜单设为快捷值或 1 到 99 的自定义值，整季模式必须写入目标 `season_number`，到期计划可设为不自动到期或 7/30/90 天后到期。后台维护只能通过 `startVideoCacheJob` 投递现有视频缓存任务，不能新增第二套下载器，不能修改 `MediaItem.filePath`，也不能写回、移动或改名用户媒体文件；维护时必须跳过未启用、暂停中或已到期订阅，排除已缓存和已在任务中心排队的条目，并在 `localNetworkOnly` 策略下只允许 localhost、`.local` 和私有网段地址。过期订阅清理只能删除 `video_offline_subscriptions` 内部规则，不能删除已缓存视频。`metadata_correction_history` 只记录 MediaLIB 内部索引覆盖的字段旧值/新值、来源和撤销时间，详情页撤销最近一次覆盖，设置页元数据历史按批次撤销；用户显式打开“写入文件”的音乐标签写回仍按原服务单独处理，不能承诺文件级撤销。`remote_connector_accounts` 只保存连接器账号的非凭据元数据和能力说明，token/密码继续走受限凭据存储；当前 Emby/Jellyfin 登录同步和 Plex Token 直连同步都会写入本地账号记录，iCloud 仍为模型预留。`sync_conflicts` 承载字段级冲突与解决状态，设置页冲突队列可记录保留本地、采用远端、合并、都保留或忽略；采用远端会把 `watched`、`watchlist`、`favorite` 和 `user_rating` 写入 MediaLIB 内部索引，评级冲突只更新用户评级字段，不覆盖资料源评分；Trakt watched/watchlist 冲突选择保留本地会写回 Trakt；Plex/Jellyfin/Emby 的更多远端写回、评分冲突来源接入和多字段合并仍必须由后续连接器执行；旧 `local_user_profiles` / `profile_media_state` 表仅作为 v18 历史兼容保留，当前不接入 顺序，不保存媒体路径，不复制或移动用户媒体文件，并通过外键在条目删除时级联移出集合；首页发布也只读取本地公开视频与远程服务器顶层视频，不把保险库内容混入总览。`media_items.rating` 始终表示 TMDB/Emby/Jellyfin/Plex 等资料源 0-10 分，`user_rating` 表示用户 1-5 星评级；迁移和元数据更新只在 `user_rating` 为空时从资料评分折算初始评级，之后 `updateRating` 只写 `user_rating`，`updateMetadata(rating:)` 不能覆盖已有用户评级。已有数据库升级前、手动恢复前都通过 SQLite backup API 创建一致性快照，不能直接复制 WAL 模式下的数据库文件。恢复前必须校验 schema 版本和 `integrity_check`，恢复时取消未落盘队列写入并只替换 MediaLIB 内部索引，不触碰用户媒体文件。`media_items.play_count` 记录音乐播放次数，扫描/upsert 保留已有播放次数；歌曲播放入口递增该字段，最多播放排序和右键/页头重置只更新 MediaLIB 索引，不改动用户音乐文件。媒体源扫描不要回到“开始扫描先删除整个 source”的路径；当前通过导入成功 ID 集合和临时 keep 表在完整成功后清理旧条目，扫描取消或存在错误时保留旧索引。来源路径归属和远程来源删除必须使用精确匹配或 `sourceRoot/` 子路径边界，调用前缀入口时会归一化尾部 `/`；不要用裸 `hasPrefix` 判断来源，否则 `/Media/A` 会误匹配 `/Media/Anime`，`emby://host/source` 会误匹配 `emby://host/source2`，`jellyfin://host/source` 会误匹配 `jellyfin://host/source2`，`plex://host/source` 也会误匹配 `plex://host/source2`。
- 数据库前缀匹配补充：落到 SQL `LIKE` 时必须转义 `%`、`_` 和反斜杠并显式使用 `ESCAPE`，否则带特殊字符的来源路径、目录名或搜索词会被当作通配符，误匹配相邻内部索引。

- 本地档案：功能已移除。不要恢复侧栏档案菜单、设置页档案管理、儿童档案或每档案状态覆盖；保险库隐私继续只由保险库 PIN/Touch ID 和锁定状态控制。旧 `local_user_profiles` / `profile_media_state` 表只为历史库兼容保留，不能作为当前功能入口。

- 片库健康：`AppState.rebuildDerivedItemCaches()` 在单次纯内存遍历中生成重复标题组和核心元数据缺口；保险库私密 ID 必须先从 `.privateCollection` 根节点沿 `childrenByParentID` 向下传播，覆盖“保险库 -> 剧集 -> 单集”等多层后代，再用于公共统计、健康清理、正在观看、Trakt 同步和侧栏分区过滤，不能退回只判断直接父级。本地文件缺失、远程播放路径缺失与来源离线检查继续由 `scheduleFileHealthRefresh()` 在 utility 后台任务中完成，SwiftUI `body` 不得同步调用 `FileManager`。本地非音乐条目只有在来源仍参与健康检查、文件路径非空且文件不存在时进入失效路径；Emby/Jellyfin/Plex 等远程媒体服务器非音乐条目在来源仍参与健康检查且 `filePath` 为空时进入失效播放路径。健康中心使用这些派生缓存和惰性静态玻璃列表，安全可清理条目预计算为 ID 集合，逐行判断保持 O(1)。失效文件/播放路径和疑似重复项的界面详情仍要按 `AppState.isPrivateItem` 做锁定态兜底隐藏，远程缺失播放路径只显示“远程播放路径未记录”，不能泄露 token 或服务端 URL。健康中心“一键补充”会把视频和音乐缺口加入任务中心，仅补充缺失字段、不覆盖已有数据；剧集/动漫视频补充复用 TMDB 多查询宽松匹配链路，音乐补充使用独立音乐匹配宽容度。失效索引清理必须显式确认，只能调用 `MediaRepository.deleteItems(ids:)` 删除内部索引；离线来源条目、锁定保险库条目与用户媒体文件都不得清理或暴露。
- 增量扫描与任务中心：`LocalFileEventMonitor` 只监听开启自动扫描、可访问且 `sourceKind == .local` 的本机来源，使用 FSEvents file events 并在 `AppState` 合并 1.2 秒内的变化。普通文件事件进入 `MediaScanner.scanChanges`，只导入/删除来源目录内部受影响路径；缺失路径若是媒体文件按单文件清理，若是旁路元数据文件则刷新同目录媒体，其余缺失路径都按目录删除候选处理，不能再用 `pathExtension.isEmpty` 判断目录，否则 `Show.Name` 这类带点号目录会残留索引；目录删除/重命名、root change 或 event dropped 必须降级为完整来源扫描。SMB/FTP/NAS 不得使用 FSEvents 删除索引，继续使用周期扫描兜底。增量删除前来源必须可访问，删除最后一集后可清理没有子项的本地系列父索引。删除媒体源时应调用来源边界清理，只删除精确 `source_path` 或 `source/` 子路径的 MediaLIB 内部索引，不能误伤相邻来源，也不能只删精确来源导致历史子路径残留。`BackgroundTaskSnapshot` 通过 `Codable` 持久化到 Application Support 的 `MediaLib/BackgroundTasks.json`，统一展示完整扫描、增量扫描、远程服务器同步、封面预热、元数据补充、视频缓存、章节图和一键清理；重启时仍处于 active 的任务必须标记 failed，避免假装继续运行。任务中心进度只按受控阈值发布，保险库任务始终隐藏条目名、来源路径、文件名和具体错误对象；`hidesDetail == true` 的任务必须使用通用 `kind.title`、空 detail、通用浮窗/系统通知/日志文案。章节图任务由 `AppState.keyframeStoryboardTasks` 跟踪并支持取消，进度只按 bucket 批次发布，不能新增高频计时器或在播放线程抽帧。视频缓存暂停必须走 `VideoCacheDownloadController.pause()` 的 `URLSessionDownloadTask.cancel(byProducingResumeData:)` 断点数据，不允许只把 UI 标为暂停；`pausing` 状态用于等待 resume data 稳定，继续按钮只能在 `.paused` 后展示。`VideoCacheDownloadController.Progress` 是累计口径，继续下载时必须保留 resume data 字节数、`lastProgress` 与预期总大小，兼容系统回调本轮字节或累计字节两种情况，不能让 UI 在续传后回到 0%。完成、失败、取消和数据库恢复都要 `invalidate()` 下载 controller 并取消 worker，避免后台下载或 delegate session 残留。一键清理只能触碰 MediaLIB 内部 `VideoCache`、空 artwork 缓存目录和任务历史，不得遍历或修改用户媒体源。视频缓存容量上限由 `AppSettings.videoCacheSizeLimitGB` 控制，0 表示不限制；容量回收只能删除 `VideoCacheManifest.json` 登记的缓存视频及其同名字幕旁路文件，优先保留最近播放条目，回收已看或较久未播放的超限缓存。设置页的占用展示走清单轻量估算，真正回收前再做完整文件尺寸核算，避免滚动设置页时同步枚举大量缓存文件。
- 媒体源页的添加入口必须保持单一“添加媒体源…”按钮，并通过 `AddMediaSourceWizardSheet` 的分步向导选择本地目录、网络设备、Emby、Jellyfin 或 Plex；不要把 Emby/Jellyfin/Plex 登录、网络设备或本地分类重新拆成工具条上的多个添加按钮。媒体源添加后不要再弹与扫描流程无关的特殊提示；保险库来源也只应通过正常扫描任务、任务中心和安全标签反馈状态。重复来源提示必须走统一安全文案，保险库类型不能在“已存在”弹窗中显示目录名或路径；网络挂载来源也要按挂载目录去重，避免重复保存同一 SMB/FTP/FTPS 目录。代码注释应说明性能原因、隐私边界或业务约束，不使用内部排期编号、临时任务编号或无法长期维护的口吻。
- 浮窗通知：轻量结果、任务加入/完成/失败/取消、用户主动切换喜欢/想看/已看、评级和删除播放记录后的状态反馈、一次性页面提示统一走 `AppState.showFloatingNotice` / `showInterfaceTipOnce`，由 `ContentView` 顶部中央的 `FloatingNoticeStack` 呈现胶囊玻璃通知。通知应保持短标题、可选说明、自动消失和手动关闭；胶囊宽度必须按标题与说明的真实文本宽度计算，短提示收窄，长片名/路径/任务详情在最大宽度内居中换行。任务类通知使用短标题，例如“视频缓存已加入任务”，具体对象放进说明行，避免把完整任务名塞进标题撑宽浮窗。页面提示必须是用户能直接理解的使用建议，并通过 `UserDefaults` 记录为首次进入后不再展示；保险库锁定时不得在通知说明里显示条目标题、来源名、路径或文件名。不要用浮窗承载需要确认的危险操作，也不要把已有确认弹窗替换成无确认浮窗。新增通知动画应走 `AppMotion.notice` 并尊重 Reduce Motion；通知、页面提示、`AppInfoNote` 弹窗提示卡、`AppInlineNoticeLabel` 行内说明和播放器 toast 的图标颜色统一使用 `AppColors.selectedGlassTint`，错误/警告等仅通过图标形状和文字表达类型。
- 首页与详情页推荐：总览推荐只能基于 `AppState` 已有派生缓存和 `MediaItem` 内存字段计算，不在 SwiftUI `body` 中读盘、联网或新增高频任务。推荐画像使用看过、继续观看、喜欢、想看、用户评级、资料源评分、题材标签和媒体类型等现有信号，优先展示评分较高且未看完的剧集/系列；`overviewSnapshotKey` 必须覆盖 `libraryRevision`、`favoriteRevision`、`watchlistRevision`、`ratingRevision`、`videoCacheRevision`、`AppSettings.watchedThreshold` 和日期种子，避免用户意图或已看阈值变化后看板不刷新。新增首页看板应复用 `HomeOverviewBoard` 的横向条、滚动期 hover 抑制和主布局切换暂停逻辑；空的上下文看板不要强行展示，已发布集合仍只读取公开视频与远程服务器顶层视频，不混入保险库。详情页 TMDB 相关推荐仍按需拉取外部 similar，不新增后台预取；本地交叉匹配排序应优先已入库、高评分、题材重合、用户评级、想看/喜欢和未看状态，且保险库锁定时不得把私密条目标成“在库”或泄露标题。
- 视频用户意图：`MediaItem.favorite` 表示喜欢，并继续承担 Emby/Jellyfin 收藏同步；Plex 第一阶段的喜欢状态只保存在 MediaLIB 本机，不写回 Plex 服务端。`MediaItem.watchlist` 表示本机想看计划，扫描和 `replaceRemoteItems` 刷新都必须保留。手动或批量标记视频已看、同步冲突采用远端已看时，应通过 `MediaRepository.markWatched(... clearWatchlistWhenWatched: true)` 清理本机想看；播放进度自动达阈值的 `updatePlayback` 不触发该清理，避免后台进度写入隐式改变用户计划。所有“已看/未看/正在观看”筛选、统计和整季按钮都必须读取 `AppSettings.watchedThreshold`，不要硬编码 90%；阈值保存变化后要重建 `AppState` 派生缓存并刷新库页面快照 key。手动取消已看必须同步清空播放位置、完成进度和最近播放时间，否则 `watched || playProgress >= watchedThreshold` 的页面筛选仍会把条目判为已看；取消已看不能清除用户的想看计划。手动取消已看或清除播放记录后，`AppState` 必须记录清除前的条目修订并忽略该修订及更旧播放器快照的延迟 `updatePlayback`，避免关闭窗口或切集回调用旧高进度重新写回“已看/正在观看”。手动标记已看时，如果条目可表达为 Trakt watchlist，还要同步移出 Trakt 想看。左侧本地“视频”目录只展示本地公开视频；EMBY / Jellyfin / Plex 顶层视频只进入各自独立目录，其中本机想看显示在对应远程目录下的“想看”；保险库喜欢/想看只在解锁后的保险库页面生效，不得把保险库条目或其后代混入本地“视频 > 想看/喜欢”。首页是独立口径，可使用 `homeVideoItems` 汇总本地公开视频 + 远程服务器顶层视频，但仍不得包含保险库。`VideoSmartCollectionRepository` 保存基础范围和 `rules_json` 复合规则，`VideoSmartCollection.matches` 统一动态求值；媒体类型始终作为智能集合基础边界，AND/OR 只作用于状态、最近加入、年份、评分、评级、题材和来源等条件。智能集合不得复制媒体 ID、移动文件或混入保险库与剧集子项。
- Emby/Jellyfin 状态同步：`AppState.withValidEmbySession` 统一处理凭据加载、401/403 自动重新认证和新 token 保存；播放前先轻量校验会话并刷新流 URL 中的 `api_key`。`MpvPlayerController` 只在播放开始、暂停/恢复、停止和每 15 秒节流点发布 `PlayerPlaybackReport`，不要把远程请求接入 0.18/0.25 秒进度热路径。远程库写入必须使用 `MediaRepository.replaceRemoteItems` 的单事务路径，并采用服务端返回的用户状态。Emby/Jellyfin/远程视频的“缓存到本地”由 `VideoOfflineCacheStore` 管理：默认写入 `Caches/MediaLib/VideoCache`，用户可通过 `AppSettings.videoCacheDirectoryPath` 选择自定义根目录，实际文件仍统一落在该根目录下的 `VideoCache` 子目录；清单始终保存在 Application Support 的 `VideoCacheManifest.json`。播放和外部打开可优先使用有效缓存副本，但数据库 `MediaItem.filePath` 仍保持远程源路径，不移动、不改名、不写回用户媒体或服务端文件。首页“离线”标签只从 `homeVideoItems` 的公开视频/远程服务器顶层视频中筛选已缓存范围，并随 `videoCacheRevision` 刷新；保险库内容仍只在解锁后的保险库页面显示。删除缓存只能删除清单登记的本地缓存视频与同名旁路字幕；字幕副文件只允许匹配完整视频 base 或 `<video-base>.` 边界，不能用裸字符串前缀，避免删除 `abcde.zh.srt` 这类相邻缓存。系列删除通过子集缓存 ID 扩展作用域，单集删除只传入该集。重新缓存同一条目前必须调用旧缓存清理并用任务内 `cleanedItemIDs` 防止暂停/继续后重复删除。原画直连缓存保留容器内音轨；远程服务器文本字幕通过 `EmbyService.subtitleStreams` / `downloadSubtitle` 尽量写成同名 `srt/ass/ssa/vtt` 旁路字幕，字幕失败只记 warning，不让视频缓存失败。非原画转码档位的音轨/字幕能力以服务端转码输出为准。缓存清晰度必须复用 `RemoteVideoQualityPlanner` 的媒体服务器档位规则，不要把挂载 NAS 的播放端降采样伪装成可降低网络读取的下载清晰度。系列自动缓存订阅由 `VideoOfflineSubscriptionRepository` 持久化，并由 `scheduleVideoOfflineSubscriptionMaintenance` 低频维护；候选集必须排除已缓存和已在任务中心排队的条目，`.nextUnwatched` 按保存的 `episodeLimit` 维护当前未看窗口内缺失的缓存，不能跳过已缓存剧集后无限向后扩张窗口；`.season` 只处理保存的 `seasonNumber` 对应季，`localNetworkOnly` 不能排入公网 URL，`.wifiOnly` 必须读取 `NWPathMonitor` 发布的当前 Wi-Fi 状态且只控制新自动缓存任务排入；`scheduleVideoOfflineSubscriptionExpirationCheck` 只负责到期后关闭订阅规则，不删除 `VideoCacheManifest.json` 中的缓存条目。
- Plex 状态同步：`PlexService` 使用 Plex XML API 直连服务器，凭据只保存服务器 URL、Token 和 machineIdentifier，不保存用户名密码。`withValidPlexSession` 不做自动 token 刷新，401/403 必须提示重新连接。播放前只刷新 URL 中的 `X-Plex-Token`；播放状态写回 `/:/progress`，已观看写回 `/:/scrobble` / `/:/unscrobble`，Plex 收藏/评分/字幕 sidecar/转码质量仍未接入。Plex 条目使用 `sourcePath + ratingKey` 生成内部稳定 ID，刷新只替换 MediaLIB 内部索引，不修改 Plex 媒体文件或服务端元数据。
- Trakt 状态同步：`TraktService` 使用设备码授权，token 保存在本机设置；连接成功后写入 `remote_connector_accounts` 的 syncOnly 账号记录。默认路径仍是本机已看/想看推送到 Trakt；本机手动标记已看会同步 history，并在本机想看被自动清理时同步 remove watchlist。“从 Trakt 导入”只拉取 watched movies/shows 与 watchlist，并对已匹配 TMDB 的公开视频生成 `sync_conflicts`，不直接覆盖 `media_items`。冲突 ID 必须按 `trakt-<mediaID>-<field>` 稳定生成，重复导入更新同一条 pending 冲突；保险库内容和未匹配 TMDB 的条目不得导入或推送。当前 Trakt 导入只生成 watched/watchlist；采用远端会在冲突队列中写入本机 `media_items` 的已看/想看/喜欢或用户评级状态，采用远端已看时会清理本机想看但不额外反推 Trakt watchlist，保留本地会通过 Trakt sync API 写回 watched/watchlist。`TraktSyncPayloadBuilder` 位于 `MediaLibCore`，用于生成 movies/shows/seasons/episodes 载荷并由 `MediaLibChecks` 覆盖排序去重；Trakt 评分同步、失败恢复和更多字段合并后续补。
- 用户歌单与队列：`MusicPlaylist` / `MusicPlaylistRepository` 使用 `music_playlists` 和 `music_playlist_items` 持久化用户歌单；`MusicQueueSnapshot` / `MusicQueueRepository` 使用 `music_queue_state` 和 `music_queue_items` 持久化当前队列顺序、随机和循环状态。两者都只保存 MediaLIB 内部 `media_items.id`，不移动、不改名、不复制用户音乐文件。队列仍由 `AppState.musicQueue` 驱动，变更经短防抖后台写入；启动恢复队列但不自动播放。收藏不再作为音乐侧栏独立页面展示，而是由 `MusicLibraryView` 生成不可删除的置顶虚拟歌单，移出歌曲时更新 `MediaItem.favorite`，不要把该虚拟歌单写入 `music_playlists`。
- 打包：SwiftPM + 手工 `.app` Bundle + `hdiutil` DMG。打包脚本会在本机存在 Homebrew `ffmpeg` 时把可执行文件和依赖复制进 App，用于 AVFoundation 无法处理的 MKV 截帧封面兜底；如果 Homebrew 依赖链间接复制 `Python.framework`，脚本只保留动态加载所需的框架二进制和资源骨架，剔除 stdlib、tests、docs、headers、bin/include/lib/share 等无用目录，避免 App 包体积和潜在加载压力失控。脚本会清理普通文件和符号链接自身的扩展属性并以 `/private/tmp/MediaLib-package/MediaLIB.app` 作为严格签名校验对象；`dist/MediaLIB.app` 位于 Documents/FileProvider 管理目录时可能被系统重新写入 `com.apple.provenance`，最终交付以已校验的 `dist/MediaLib.dmg` 为准。

## 扫描规则

- 经典目录 `TV Shows/剧名/Season 01/S01E01.mkv` 优先使用“剧名”目录作为系列身份。
- 如果存在每集 `.nfo`，其 `<title>` 不会覆盖系列名，避免同一剧集被拆成多个系列。
- 媒体源类型会影响分类：动漫源进入动漫栏，纪录片源进入纪录片栏，综艺源进入综艺/系列类型，其他源进入其他栏，保险库源进入上锁分类。
- 自动识别媒体源需要同时穿透识别视频和音频；音频文件扫描使用低大小阈值，避免普通歌曲被视频默认的 50MB 阈值跳过。
- 扫描器必须通过 `FilenameParser.isSidecarMetadataFile` 统一排除 `.lrc`、`.txt`、字幕、本地图片、`.cue`、`.nfo` 等旁路元数据文件，只把规范化后的普通媒体文件放入导入队列，并用规范化路径去重；增量扫描遇到这些旁路文件变化时应刷新同目录媒体，但不能把旁路文件自身导入。音乐导入时会清理同一 `file_path` 的旧重复行，避免同一个物理文件在歌曲列表出现多条。
- 音乐扫描只使用音频标签中的 `album`，不要从文件夹名推断整张专辑；无内嵌封面时先留给默认图标/默认封面和后续联网补全，不要用同目录其他歌曲或 `cover/folder/poster` 串用。音乐默认封面由专用音乐图形绘制，使用当前主题派生的玻璃渐变、唱片/音符和波形元素，不要复用视频播放卡图形，也不要退回固定蓝青色。
- 音乐歌词只匹配同名 `.lrc` / `.txt`；不要读取目录级 `lyrics.lrc` / `lyrics.txt` 作为整个文件夹的共享歌词。
- “音乐元数据获取”工作台的文件写回只能处理本地、可写、受支持的音频格式；远程 URL、不可写路径和不支持格式必须逐条失败并保留原文件。封面嵌入失败时可以退回只写文字标签，不要因为封面失败而破坏整首歌。
- Emby/Jellyfin 使用 API 登录同步，不再按匿名分享链接导入。`RemoteCredentialStore` 将 token/网络凭据写入 Application Support 的 `MediaLib/Credentials` 文件，基础源 `sourcePath` 使用 `emby://host/sourceID` 或 `jellyfin://host/sourceID`，媒体源列表固定显示 `EMBY` / `Jellyfin` 且不支持手动重分类。连接成功后还会写入 `remote_connector_accounts` 的非凭据账号记录；删除失败来源或删除媒体源时必须按 `sourceID` 清理对应账号。同步时先读取 `Users/{userId}/Views`，再按每个 View 的 ParentId 递归拉取 Movie/Series/Episode/Audio；导入条目的 `sourcePath` 会扩展为 `<scheme>://host/sourceID/library/<viewID>/type/<collectionType>/name/<encodedName>`，`externalID` 保存服务端 ItemId，刷新时用 provider 前缀稳定 ID 更新内部条目。`media_sources.selected_emby_library_ids` 字段名为兼容旧库保留，但现在承载 Emby/Jellyfin 服务器 ViewId 列表。远程条目必须保持独立命名空间，不要放回本地 `cachedTopLevelItems` / `cachedMusicTracks` 或普通视频/音乐侧栏；侧栏远程目录只显示有条目的通用分类和服务端媒体库分类。删除或刷新远程源要按 sourcePath 前缀清理旧条目，不能只匹配基础源路径；旧 Keychain 项不迁移也不清理，避免更新后触发钥匙串 ACL 弹窗。
- Emby 剧集需要用服务端 `SeriesId` 作为 Episode 的 `parentID`，不要只用 Episode 的 `ParentId`，因为后者通常指向 Season，会导致 MediaLIB 的剧集详情页找不到子集。如果某些 View 只返回 Episode 而没有返回 Series，`EmbyService` 会用 `SeriesId/SeriesName` 合成一个剧集父项，保障剧集列表仍可展示。Emby 远程播放地址由 `MediaSources` 辅助生成：优先使用 `MediaSourceId` 和 `Container` 生成 `Videos|Audio/{ItemId}/stream.<container>?Static=true&MediaSourceId=...&DeviceId=...&api_key=...`；同步时尽量读取 `MediaStreams` 的分辨率、音视频编码和文件大小，让远程条目拥有接近本地条目的元信息。http/https 条目必须继续跳过本地文件存在性检查，直接交给 mpv 或系统播放器；系统播放器模式要优先打开已安装的视频播放器应用，避免 `NSWorkspace` 把远程 URL 交给浏览器。详情页不要直接显示带 token 的完整 URL。
- Emby/Jellyfin 登录提交后由弹窗立即 `dismiss()`，`AppState.connectEmbyServer` / `connectJellyfinServer` 在后台维护认证、任务中心进度、全局结果对话框与首次失败回滚；不要让登录弹窗等待网络返回。详情页播放入口应传入 `preserveSelection: true`，关闭播放器后继续保留当前详情上下文。
- SMB/FTP 网络源由 macOS 负责登录和挂载，用户选择挂载目录后复用 `MediaScanner` 扫描；凭据只用于记录来源和后续扩展，不在数据库明文显示。扫描前若已保存的网络源挂载目录不可达，`AppState` 可通过 `NSWorkspace.shared.open` 打开保存的 `smb://` / `ftp://` / `ftps://` 来源 URL 触发系统重新挂载，并轮询原本地挂载目录；媒体源列表的“重新挂载”按钮调用同一套 `remountNetworkSource` 流程，成功后清理离线源缓存并刷新页面。不要为了自动重连手写 SMB/FTP 目录解析器或移动用户文件。
- `AppSettings.automaticScanInterval` 控制定时扫描；`AppState` 只扫描 `autoScan == true`、非 Emby 且当前真实可访问的路径。扫描入口要使用实时可达性，避免 NAS 曾经离线后缓存状态阻止重连后的手动/自动扫描。
- 保险库内容通过 `AppState.isPrivateItem` 从普通首页、最近播放、收藏、健康提示等入口中过滤。
- 剧集系列缺少本地 poster 时，会复用首个成功生成的剧集封面，让系列海报墙不再空白。
- 横版抽帧封面按图片真实比例显示，UI 层通过 `ArtworkMetrics` 对极端比例做约束。

## 播放器现状

- 视频普通播放入口在 `AppState.presentBuiltInPlayer`，当前只设置 `activePlayerItem`，由 `VideoPlayerWindowPresenter` 打开 `PlayerView.swift` 的 SwiftUI 独立窗口。
- `BundledMpvPlayerService.swift` 已移除，不要再恢复完整 mpv 可执行文件窗口作为默认入口。
- `AppSettings.videoPlayerPreferredWidth` 仍保存播放器目标内容宽度，但设置页通过 `VideoWindowSizing.screenWidthRatio` 显示为当前屏幕可用宽度百分比，限制在 45% 到 100%；写回时再用 `preferredWidth(forScreenWidthRatio:)` 转成点数。打开新视频时会以发起播放的主窗口所在屏幕为准，先结合索引视频比例计算透明标题栏 `ImmersivePlayerWindow` 的 frame，mpv 准备好后再根据 `dwidth/dheight` 或 `width/height` 回传的真实显示比例校正窗口，若超过屏幕可用区域则按接近全屏的可用区域等比缩小。设置为 100% 时，`VideoWindowSizing.usesFullScreenWidth` 会允许 `preferredPlayerSize` 直接使用当前屏幕 `visibleFrame.width`，不要再额外乘 0.985 或类似安全系数；非 100% 设置仍按原有安全边界收敛。
- 主窗口内容最小尺寸在 `MainWindowToolbarVisibilityGuard.minimumContentSize` 固定为 1088x720，由 220pt 侧栏、约 848pt 主内容区、页头操作区和底部音乐栏安全高度推导；需要同时写入 `contentMinSize` 和 frame 级 `minSize`，并在 `NSWindow.didResizeNotification` 中兜底回弹，防止 AppKit/SwiftUI 场景恢复后仍可手动拖到红绿灯大小。不要在单个页面用更大的隐式最小宽高把主窗口撑开，也不要给主窗口设置会破坏最大化/全屏的 `contentMaxSize`。视频播放器最小内容尺寸由 `VideoWindowSizing.minimumControlSafeWidth` / `minimumControlSafeHeight` 和当前视频比例共同计算，并同时写入 `contentMinSize` 与 `minSize`，`VideoPlayerWindowPresenter.Coordinator.windowWillResize` 还要拦截过小的拖拽提案，保证底部 596pt 控制条、18pt 左右安全边、锁定按钮和加载层在缩小时不重叠。
- `LibMpvClient` 使用 `mpv_render_context_create` / `mpv_render_context_render` 官方 render API，将视频帧绘制到 `MpvOpenGLView`；不要改回 `wid` 或完整 mpv 可执行文件窗口路径，否则在部分 macOS 环境会再次出现视频窗口和控制窗口分离、Dock 多出图标的问题。
- `LibMpvClient` 会为弱网播放设置 mpv 内存缓存、预读和 cache-pause 相关选项；这些参数应保持可容错设置，不能因为某个 libmpv 构建不支持某项参数而阻断播放器启动。
- `LibMpvClient.setString` 调用 `mpv_set_property` 时必须按 `MPV_FORMAT_STRING` 传入 C 字符串指针的地址，不要改成直接传 `value.withCString` 得到的字符缓冲，否则 mpv 会把字符串字节误读成地址，播放器启动应用画面比例、去隔行、旋转、循环或滤镜属性时会崩溃。
- 当前 `NSOpenGLView` / `NSOpenGLContext` / OpenGL 调用会触发 macOS 10.14 起的弃用警告，Apple 官方建议迁移到 Metal/MetalKit；但 MediaLIB 的视频窗口现在依赖 libmpv render API + 应用内 OpenGL 视图来保持单窗口控制层、系统圆角和原生红黄绿按钮。本项目不要把它当作普通组件替换，未来应单独设计和验证 Metal/libmpv 渲染路径后再迁移。
- 视频控制层由 SwiftUI 自绘：播放器内部使用自动隐藏的底部紧凑液态玻璃悬浮条，窗口层保留 macOS 原生红黄绿按钮；红黄绿按钮由 `PlayerWindowChromeVisibility` 跟随控制层淡入淡出，形成接近系统播放器的无边框观感。`VideoControlBrightnessSampler` 只在打开视频时对海报做小尺寸亮度取样，`VideoControlPalette` 统一驱动控制条、时间文本、图标、锁定按钮、进度和播放器玻璃底的黑/白高对比内容色；不要在播放中逐帧采样视频画面，也不要把控制层文字固定成某一种颜色导致暗/亮视频下不可读。视频音量、音乐音量和键盘音量微调统一通过 `PerceptualVolumeScale` 做感知曲线映射，UI 滑条可以是感知值，播放器和设置保存仍保持 0...1 线性音量。
- 视频播放器专属设置分两层：`PlayerControlsBar` 的齿轮弹层只保留播放中常用的窗口宽度、窗口置顶、预览图模式和“更多设置”入口；`PlayerAdvancedSettingsOverlay` 在播放器中央承载完整设置，包括视频播放方式、系统播放器路径、默认倍速、快进/快退间隔、记忆进度、自动下一集、自动标记已看、音频延迟、字幕延迟、字幕大小、字幕位置、画面比例、黑边裁切、去隔行、画面旋转、单片循环、A-B 循环、播放信息、快捷键和触控板手势。所有变更都直接写入 `AppSettings` 并调用 `AppState.saveSettings()`，并通过 `MpvPlayerController` 立即写入当前 libmpv 实例；不要让完整设置窗口订阅控制器高频进度。播放信息弹层只能展示用户可理解的媒体摘要、当前轨道、清晰度、倍速、音量和调校状态，不显示本地文件路径、保险库路径、扫描路径或带 token 的远程 URL。窗口宽度在播放器内调整时可以同步 resize 当前 `ImmersivePlayerWindow`，但必须保持非全屏保护、屏幕可见区域限制和现有 `contentAspectRatio` 逻辑；窗口置顶由 `AppSettings.videoPlayerAlwaysOnTop` 持久化，并在窗口创建、重复 present 和当前窗口切换时同步 `NSWindow.Level`。设置页只保留视频默认启动方式作为安全回切入口，以及视频缓存位置/容量这类库级设置；不要把窗口宽度、预览图模式等播放器行为项再放回设置页主流程。
- 视频画质入口按钮只显示“画质”，具体分辨率/码率档位留在 `PlayerQualityPopover` 列表中展示；控制条、弹层和播放/暂停按钮使用较轻的播放器玻璃材质，不要恢复深黑大阴影或把播放/暂停按钮改回圆形。
- 视频关闭、窗口红黄绿关闭和上下集切换必须先保存进度并调用播放器 `teardown()`；`LibMpvClient.stopPlayback()` 会静音、暂停并 stop mpv，避免关闭后音频残留。OpenGL 绘制首帧前先清黑，避免切集时露出未初始化蓝屏。
- 视频窗口是透明标题栏的沉浸式标准窗口，不要改回纯 `.borderless`；窗口背景需要保持 `isOpaque = false` 和 `.clear`，让系统圆角裁切和原生按钮正常显示。根 `PlayerView` 不应设置固定最小高度，否则超宽视频会被 SwiftUI 撑高并出现上下黑边。
- 透明视频交互层需要区分点击与拖动，拖动时调用 `NSWindow.performDrag(with:)` 移动整个播放器窗口，播放器窗口类需要允许成为 key/main window 以接收快捷键。
- `KeyCaptureView` 负责播放器快捷键捕获，但不再硬编码动作表；默认键位和用户覆盖保存在 `AppSettings.videoKeyboardShortcuts`，通过 `VideoPlayerShortcutAction` 解析。默认保留 Space/K 播放暂停，方向键、Shift/Option 方向键、J/L/PageUp/PageDown/Home/End/数字键处理跳转，M 静音，F/Return/Cmd-F 全屏，Esc/Cmd-W/Q 关闭或退出，`[`/`]`/`\` 调整倍速，`,`/`.` 逐帧，Z/Shift-Z 调整字幕延迟，Control--/Control-= 调整音频延迟，G/Shift-G 调整字幕大小，R/Shift-R 调整字幕位置，D 切换去隔行，C/V/A 处理字幕和音轨，B/Shift-B 设置或清除 A-B 循环，I 打开播放信息。播放信息弹层打开时，Esc 或 I 应先关闭弹层，不应直接关闭播放器窗口。`VideoShortcutRecorderView` 只在更多设置窗口记录新组合键；重复组合键应通过 `AppSettings.setVideoKeyboardShortcuts` 自动从旧动作移出。快捷键比较必须先规范化 AppKit 为 Esc、方向键、Home/Page 等非文本键塞入的控制字符和私有功能键字符，只保留稳定的 `keyCode + modifiers`，避免默认快捷键和真实事件对不上。未匹配的按键必须交还 AppKit，避免吞掉系统 Command 组合。
- `PlayerInteractionOverlay` 负责播放器点击、拖动和触控板手势。触控板横向精确滚动按 `AppSettings.videoTrackpadHorizontalSeekEnabled` 与灵敏度换算为小步 seek，竖向精确滚动按 `videoTrackpadVerticalVolumeEnabled` 调整音量，捏合全屏由 `videoTrackpadPinchFullscreenEnabled` 控制且默认关闭；普通鼠标滚轮不要触发视频跳转。
- 播放画面右键菜单保持精简，只保留从头播放、截图和系统播放器打开这类上下文动作。字幕、音轨、清晰度、倍速、章节、全屏、上一集/下一集、窗口置顶、单片循环、A-B 循环、播放信息和完整设置已经在控制条或齿轮弹层有入口，不要再重复加回右键菜单。
- 视频窗口打开后通过 `centerWindow(_:on:)` 按发起播放窗口所在屏幕的 `visibleFrame` 居中，并在 SwiftUI 内容挂载后再次校正位置，不要只依赖 `NSWindow.center()` 或 `NSApp.keyWindow`。
- 视频控制层自动隐藏计时器放在 `PlayerControlsAutoHideCoordinator`，控制条常驻视图层级并切换透明度/命中测试；不要把每次鼠标移动都会变化的 Task 存回 SwiftUI `@State`，也不要在隐藏时移除控制条视图，否则控制条会因频繁重绘或 hover 中断出现闪烁。
- 视频进度条悬停预览由 `VideoProgressScrubber` 管理，预览气泡必须作为独立 overlay 保持固定高度；hover 写入需要按预览帧 bucket、预览模式的最小时间间隔和最小位移阈值合并，远程/挂载网络视频以及流畅优先模式使用更粗的预览 bucket。拖动进度条时只更新本地 draft/时间气泡，不能强制每个鼠标事件都请求预览图；内存未命中时要短暂等待鼠标稳定，后台抽帧队列过忙时应跳过当前预览而不是排队抢占播放资源。松手后再提交一次 seek，避免清晰度切换期间连续 seek 被 mpv 新时间轴回写拉回 00:00；普通 hover 不要恢复每个鼠标事件都写 `scrubberPreview` 或直接触发 AVFoundation/ffmpeg 抽帧。
- 清晰度切换不要把 `hasVideoFrame` 主动置空或长时间显示加载层。Emby 转码清晰度通过带 `StartTimeTicks` 的 URL 从当前逻辑时间续播，并维护 `playbackTimelineOffset` 让 UI 仍显示原片完整时长；用户拖回当前转码分段之前时，应重新加载同一清晰度并用目标时间生成新的 `StartTimeTicks`。切回原画时使用 `loadfile ... replace start=<resume>`，并通过短时 `enforceQualityResumeTime` 校正，尽量保留播放状态与当前画面。
- 字幕和音轨已有应用内图标菜单，可调用 libmpv 切换/关闭/添加同目录字幕；后续仍需补真实轨道列表、当前选择状态和更完整的字幕管理。
- 音乐播放器在 `MusicPlayerView.swift`，由 `ContentView` 作为主界面覆盖层承载，不再弹出独立 `NSWindow`；内置音乐播放使用单实例 AVFoundation `AVQueuePlayer`，同一个 `MpvPlayerController` 在全屏播放页和底部迷你播放器之间持续存在。
- 音乐播放页关闭会清理 `activePlayerItem` 并保存进度，最小化只切换到主界面底部迷你播放器，不销毁播放器控制器。
- 视频和音乐音量通过 `AppSettings.lastVideoVolume` / `lastMusicVolume` 分别记忆用户上次调节值；旧 `defaultVolume` 只用于迁移旧配置初值，不再出现在设置页。音乐控制条只显示音量按钮，点击后弹出滑条。
- 音乐切歌必须复用现有音频后端：`MpvPlayerController` 持有单个 `AVQueuePlayer`，本地顺序播放/队列循环的即时衔接只插入一个确定的下一首并让同一播放器自动前进；手动下一首可复用该预加载项，随机、远程资源和柔和淡入走替换当前项的稳定路径。预加载任务必须可取消，前向缓冲保持有界；单曲循环、单项队列循环和队尾再次播放必须在系统移出已结束条目后重建当前项。音乐不恢复历史进度，`configure` 和 `switchNativeAudio` 都从 0 秒开始；保存音乐播放痕迹时位置固定写 0。切歌使用播放代次校验 `AVPlayerItem`、seek 完成回调和播放结束通知，避免旧异步回调把声音和封面拖成一拍错位。隐藏的 `MusicPlaybackHost` 在响应 `item.id` 变化时必须从 `AppState.activePlayerItem` 读取当前曲目，不要直接使用 `onChange` 闭包捕获到的 `item`。切歌保存上一首播放痕迹时使用 `reloadLibrary: false` 静默写库，关闭播放器也不要触发全库 reload。
- 音乐响度与过渡：`AudioMetadataReader` 在扫描阶段读取 ReplayGain/R128 曲目/专辑增益和峰值，`MediaScanner` 写入 schema v4 可选字段；`MpvPlayerController` 只通过 `MusicLoudnessGain` 调整有效输出音量，用户音量值仍保持独立，不能重编码或修改原文件。`softFade` 仅在同一个音频控制器切到下一首后执行短时平滑淡入且不启用预加载；真正重叠交叉淡化或远程预加载要作为单独架构变更验证 AirPlay、歌词时钟、队列状态和资源占用。
- 底部迷你播放器控制条必须保持弹性宽度：由 `ContentView.musicMiniPlayerWidth` 使用窗口剩余空间和 24pt 左右安全边距计算，是否避让侧栏直接由 `NavigationSplitView` 的 `columnVisibility` 推导，侧边栏收起时拉伸、展开时缩回，并尽量向右填满可用空间；不要再用只在最小化流程里手动设置的布尔值，也不要再给底部条内部设置固定最大宽度。底部条已彻底移除快退/快进 15 秒按钮，空间不足时只压缩曲目信息区域；循环模式按钮必须位于随机播放左侧，队列按钮必须位于 AirPlay 左侧。随机播放由 `AppState.musicShuffleEnabled` 独立驱动，循环由 `AppState.musicRepeatMode` 在顺序播放/队列循环/单曲循环中切换，图标分别使用顺序箭头、repeat、repeat.1。展开页第一行顺序为喜欢、弹性进度、队列，第二行为 AirPlay、音量、上一首、播放/暂停、下一首、随机、循环模式，不显示快退/快进 15 秒按钮，循环模式不要拆成三个按钮；展开页播放/暂停按钮复用底栏蓝色胶囊主按钮。底栏主体保持克制白色厚玻璃，专辑色柔光必须裁剪在圆角内部并按按钮与封面的距离递减轮廓光，不能整条染色、溢出到底栏外，也不要恢复整条底栏实时鼠标液态光或播放时间驱动 blur。列表/设置等页面任意方向滚轮或触控板开始滚动时，`MusicMiniPlayerCollapseScrollMonitor` 可把底栏收成右侧 72pt 封面形态；完整底栏封面和收起态封面必须共享 `matchedGeometryEffect`，让收起看起来是封面向右移动，展开看起来是封面向左回到完整底栏。收起态只保留变暗封面和低开销真实频谱：`MusicMiniPresetSpectrum` 出现时才调用 `MpvPlayerController.setAudioSpectrumVisualizationActive(true)`，控制器后台用 `AudioSpectrumAnalyzer` 读取本地音频极短窗口并发布 5 段能量条，暂停时静态；不要用全宽实时鼠标光、播放时间驱动 blur 或高帧率 Timeline 模拟频谱。AirPlay 按钮使用固定深蓝玻璃色，不跟随专辑封面取色；可见 SwiftUI 按钮负责点击并通过 activation 触发隐藏原生 `AVRoutePickerView`，避免透明命中层失效。播放队列只通过底栏/展开页按钮弹出的透明玻璃弹层管理；用户手动歌单单独位于音乐侧栏“歌单”入口，不要把队列重新做成独立侧边栏页面。媒体库返回顶部按钮在音乐底栏显示时要上移但不要对底部 padding 做隐式动画，避免压到右侧关闭/队列按钮并减少列表重排掉帧。
- 底栏收起态的父层宽度必须保持完整可用宽度，不能在收起时把外层 frame 缩成 72pt；收起/展开只让内部封面在同一上层坐标系里向右贴边或向左回到完整底栏，避免滚动页面影响封面移动方向。收起态进度环颜色由 `AlbumColorPalette.progressLight/progressDark` 从专辑取色邻近色生成，浅色代表未播放、深色代表已播放。
- 音乐展开页由 `ContentView` 负责窗口级覆盖，但展开过程必须拆帧：先通过 AppKit 守卫无动画切换到 `fullSizeContentView`、透明 titlebar 和保持 toolbar safe area 稳定的沉浸 chrome，约一帧后展开 `MusicPlayerView`，播放器 overlay 遮住界面后再无动画把 `NavigationSplitView` 收到详情区；展开态必须保留 macOS 系统红黄绿按钮，但只应隐藏导航标题和侧栏展开按钮，不要切换 `NSToolbar.isVisible` 或 `toolbarStyle`。不要用 SwiftUI `.toolbar(.hidden, for: .windowToolbar)` 做这个隐藏，因为它会连系统红黄绿一起隐藏。展开播放器时 `navigationRoot` 应停止命中测试，并向下层写入 `suppressPointerHoverDuringScroll`，避免被覆盖的列表、海报和按钮继续响应鼠标 hover 或光效。音乐播放状态由 `ContentView` 持有的唯一 `MpvPlayerController` 和不可见 `MusicPlaybackHost` 管理；根视图只持有控制器，不要用 `@StateObject` 订阅播放进度，否则播放时会让主根视图周期重算。展开页只负责大播放界面，底部迷你播放器是独立 `MusicMiniPlayerBar`。最小化时先缩到底部迷你播放器，再用同一侧栏曲线恢复侧边栏，并让迷你播放器为侧边栏预留宽度。展开/收起动画只使用轻量透明度转场，封面和标题不要恢复 matched geometry；也不要让侧栏收起/恢复动画、窗口 chrome 切换和播放器转场同帧执行，否则音乐列表页展开/关闭会重新掉帧并拖慢系统动画。底部迷你播放器必须由 `ContentView` 外层窗口级 `GeometryReader` 给出整窗尺寸并在父层贴底，不要把贴底交回 `MusicPlayerView` 内部 `GeometryReader`，否则在 `NavigationSplitView` overlay 下会拿到错误高度并漂到列表中部。不要再给 `MusicPlayerView` 按歌曲 `.id` 强制重建，否则切歌会重新创建整棵播放界面并造成卡顿。堆叠布局歌词卡片高度必须限制在窗口内部，控制区不要用过大的固定最小宽度把主窗口撑高。
- 音乐展开页左上角的软件内最小化按钮必须由 `MusicExpandedLayout` 显式计算位置，并让按钮左边距与左侧控制栏左边距一致，不能放进会被封面、歌词布局或材质层挤压/遮挡的内容流，也不要依赖外层 safe area overlay。按钮使用和右侧歌词卡片一致的中性液态玻璃填充、描边、阴影和高光逻辑，形状为胶囊；它不叠专辑色 tint 或彩色浸染。展开态关闭入口不要堆在左上角，当前保留在底部迷你播放器。
- 音乐展开页左侧封面、标题和控制栏必须按窗口可用高度自适应；长标题需要限制在安全高度内并允许缩放，不能再把封面顶出窗口。底部迷你播放器要用显式底部对齐，不要随列表内容漂到中部。
- 音乐展开页封面由 `MusicExpandedArtwork` 管理播放/暂停空间感：暂停时先关闭封面扩散光，再轻微缩小并下移；播放时先恢复封面尺寸，再延迟打开光晕。设置 `AppSettings.musicAlbumCoverGlowEnabled` 关闭时，必须停用 `AlbumBlurredCoverGlowLayer` 的三层 image-based glow，只保留 `AlbumPrimarySoftCoverShadow` 的浅单色主色柔影，并让阴影随播放/暂停连续衰减；该动画只作用于封面及其近场阴影，不要驱动整页背景重建。
- 音乐界面分为两层：底层为整页专辑封面取色的厚实不透视色板背景，先用专辑色混合实底色遮住下层页面，转场期间必须保持静态。背景实现保持不透明 Metal 底色、低频封面颜料场、静态径向柔光和静态斜向光束分层；光效只随专辑取色、窗口布局和展开挂载状态变化，不再跟随播放时间、播放/暂停节奏或音频能量变化，也不要恢复 60Hz `TimelineView` / 播放时钟驱动的整屏 Canvas。光效仍以专辑封面中心为锚点扩散足够范围的径向柔光和斜向光束，视觉强度由静态 Metal 底板保持，不用脉冲来补足。不要让音乐展开页透出下层列表或普通页面，也不要用全屏系统 material/blur 去做透明背景。第二层左侧放专辑封面和液态玻璃控制栏，右侧放独立圆角液态玻璃歌词卡片。右侧卡片必须由 `MusicExpandedLayout` 显式计算 `lyricsRect`，用明确的 `x/y/width/height` 约束在剩余区域内；布局优先保证左侧封面/控制栏宽度和左边距，再让歌词卡片从右边界向左弹性收窄，并让右边距与控制栏左边距一致。歌词卡片要保留合理最小宽度，但窗口较小时优先保证左侧控制栏有足够宽度。不要再用中心点加固定高度定位，也不要回到 `HStack` + padding + `.frame(maxHeight: .infinity)` 这种会在真实窗口里被撑穿的方案。圆角应接近主窗口圆角，当前为 36pt；材质需要保持透亮、带高光描边和玻璃边缘质感，不能遮挡或压缩播放控制区。展开页播放按钮和展开页进度条使用专辑取色；底部迷你播放器在缺少专辑色或取色未就绪时回退主题高亮，不要再固定系统蓝/青色。
- 音乐展开页切歌取色必须取消上一轮 `paletteLoadTask` 并校验当前歌曲 ID；不要让旧封面异步取色结果直接写回 `albumPalette`。专辑色只驱动底层 Metal 背景、封面光、歌词高亮、进度与主播放按钮；音乐控制栏、歌词卡片和收起按钮通过 `FloatingLyricsGlass` 走中性无色玻璃，不再复用专辑色 tint、彩色描边、彩色阴影或 `AlbumLightSpillOverlay`。
- 带时间戳 LRC 在 `MusicPlayerView.swift` 解析为 `TimedLyricLine`，根据播放器当前时间弹性滚动；有歌词时不要显示获取歌词按钮。第一行歌词需要通过足够的顶部占位从卡片中心出现，当前歌词行滚动锚点保持 `.center`，并柔和变色、加粗、轻微放大；不要恢复呼吸式柔光。远离当前行的歌词通过透明度和 blur 逐渐被玻璃遮挡，但用户拖动歌词区手动浏览时必须临时取消 blur。点击带时间戳歌词行应 seek 到该时间点。增强 LRC 片段时间戳使用 `SegmentedLyricFlowText` 并优先于任何估算；普通 LRC 当前行没有片段时间戳时使用 `TimedLyricLine.progress` + `LyricHighlightEstimator` 估算行内字符进度：当前行结束时间优先取下一行开始时间，最后一句用前几句平均和保守文本时长兜底；估算器必须排除空格、换行、标点和符号，中文按字符平均，英文按单词分配，句尾权重略延长，并输出原文字符 index 给 `LyricProgressWrappingText`。普通字和片段都应让未播放内容略微下沉、播放到时按 `AppMotion.lyric` 缓慢上升并轻微放大，避免一句歌词换成多行后多行同时套同一条横向渐变；换行和行切换要走柔和动画，当前 `LyricFlowLayout` 行距和当前字上浮高度都保持收敛，避免跨视觉行时产生明显纵向跳跃。普通播放行推进有轻量等待窗口，避免时钟抖动或行边界附近提前跳下一句；若视口落后真实当前行超过容忍窗口或跨过多行，必须用追赶对齐快速回到当前行。手动浏览歌词或拖动进度条后，下一次 active lyric 变化应自动退出浏览态并滚回当前播放行，避免长期错位；进度条 seek 由 `PlaybackSeekState` 驱动，`scrubbing` 和 `seeking` 先按目标时间无动画居中目标行，`settled` 只表示播放器真实落点已回读，应立即按真实 `lyricTime` 驱动当前行和逐字高亮，不能继续冻结在 `resolvedTime`。只有 `scrubbing` / `seeking` 可以使用整行 seek 高亮；`settled` 阶段必须无动画覆盖到真实行后恢复普通播放推进。seek 后的“下一句保险”只能记录下一条时间戳作为 `LyricSeekResync` 被动重同步点，真实 `lyricTime` 到达该时间戳时滚到真实当前行；不能再用 `boundaryOverrideIndex` 覆盖 active index，也不能把当前行提前推到下一句。`audioEnergy` / `precise` 后台对齐只能补充分词/逐字片段，不能移动行级 LRC 时间戳，否则会让整行落点和原始歌词标签脱节。歌词卡片里的所有滚动视图都必须隐藏滚动条，避免滚动时破坏玻璃卡片观感。
- 歌词来源 badge 只显示“原词逐字 / 音频对齐 / 估算同步”等文字，不显示“甲乙丙”或 `abc` 图标。`TimedLyricLine.activeIndex` 不应提前推进到下一行；`currentTime` 只服务进度条即时反馈，`lyricTime` 只来自播放器真实回读，不能在 seek 提交时乐观写入。LRC 解析必须保留相同时间戳行的文件顺序；80ms 内完全重复文本或真正同一句分段可以合并为一个显示块，但含假名日文原文 + 纯汉字中文翻译必须保留为独立相邻行，播放锚点优先选含假名原文，避免翻译与原文挤成同一行或翻译抢高亮。旧 `LyricSeekRecovery` / `LyricSeekBoundaryCorrection` / `lyricClockSerial` / `TimelineSeekSnapshot` 链路已删除，后续不要恢复；歌词 seek 统一通过 `MpvPlayerController.seekState` 发布拖动预览、等待落点和真实落点。`MusicTimedLyricsScrollView` 不应通过 `.id(UUID())` 在 seek 中重建 ScrollView；自动 `proxy.scrollTo` 需要用程序滚动标记屏蔽滚轮监听，避免把自动校正误判为用户浏览。歌词卡片中心舞台光必须在文字下方且裁剪在卡片内，`FloatingLyricsGlass(isLyricsCard: true)` 只用于歌词卡片，提高中心透明度时不能让文字颜色被白光冲淡。
- 音乐队列由 `AppState.musicQueue` 管理；歌曲右键应提供加入队列、下一首播放和添加到歌单，添加到歌单菜单第一项为新建歌单，后续为已有用户歌单。队列弹层需要支持清空、移除、拖动排序、整队列存入歌单和单曲存入歌单，并保持透明液态玻璃外观，不要改回白色 `List` 卡片或独立播放列表页。队列弹层必须使用 `.glassPerformanceMode(.minimal)` 和 `.preferStaticGlassSurfaces(true)`，不要在底栏或展开页弹出的队列里恢复鼠标驱动光源。队列弹层通过 `AppState.musicQueueScrollAnchorID` 仅在本次进程内记住上次滚动停留附近的歌曲，底部迷你播放器和展开播放器共用同一恢复位置；记录项仍在队列中时恢复到它，否则回到当前播放项。专辑卡片、艺术家行和歌单卡片应支持进入分组歌曲列表；播放专辑/艺术家/歌单时通过 `replaceMusicQueueAndPlay` 用该分组歌曲替换当前队列并从目标歌曲开始，不要把分组歌曲简单追加到旧队列。添加到歌单只更新 `musicPlaylists` 内存状态和歌单表，不要 bump `libraryRevision` 或刷新整库。自动下一首由音频结束通知立即触发，不要只依赖低频进度轮询。
- 队列拖拽排序期间必须暂停 `restoreQueueScroll` 和可见行锚点更新，不能因为 `musicQueue` 每次移动都把列表滚回旧锚点；跨行移动通过 `MusicQueueDragCoordinator` 节流，并用无动画 transaction 执行，保证长队列拖动可控。
- http/https 远程资源可传给 mpv；本地文件存在性检查需要跳过远程 URL。

## 性能与缓存

- `AppState.reload()` 会重建顶层媒体、保险库、音乐、Emby、parent-child 映射、首页统计、继续观看、重复标题和可见视频分组缓存；缺失文件和离线媒体源检查由后台 file health 任务异步完成并回写 `libraryRevision`。页面读取列表时应优先复用这些派生缓存，避免 SwiftUI 页面切换时重复全量过滤、排序或同步检查文件系统。
- 设置页“性能记录”开启后，`AppState.reload()` 会写入仓库读取、派生缓存重建、文件健康检查调度和总耗时，后台 file health 完成后也会记录耗时和缺失/离线数量；底层设置 key 仍沿用 `debugLoggingEnabled` 以兼容旧配置。性能记录默认关闭，不应产生额外写盘。
- 本地和远程海报都必须进入 `ArtworkImageCache`，按目标显示尺寸下采样后缓存；远程海报使用 `remoteImageAsync` 的 `URLSession` 异步路径，受 4 路并发、14 秒超时和 24MB 响应上限保护，并由 `ArtworkRemoteImageStore` 提供 in-flight 请求去重和磁盘缓存。远程响应写入用户缓存目录 `MediaLib/RemoteArtwork`，文件名是远程 URL 的 SHA-256，不能包含 Emby token 或原始 URL；磁盘缓存默认约 256MB / 1600 文件上限，按最近访问时间淘汰。不要把 `RemotePosterImage` 改回裸 `AsyncImage`、同步 `Data(contentsOf:)` 或原图级缓存。海报加载失败会进入短时缺失缓存；媒体库 reload 会统一清理缺失标记，避免 NAS 重连后相同海报路径被永久视为缺失。不要在 `LocalPosterImage` 的每个图片加载任务里逐路径清空缺失缓存，否则滚动时会反复触发 `FileManager` 检查。
- `AppState.libraryRevision` 是媒体库快照版本号，音乐页等需要本地展示快照的页面应监听它，而不是在 `body` 中反复从 `items` 推导昂贵列表。
- `ContentView` 主详情区不要用侧边栏 selection 作为 `.id` 强制重建页面；普通页面切换尽量直接换内容，不要给大列表套整页 transition/animation，避免出现一帧空白或“加载感”。
- `LibraryView` 使用 `LibrarySnapshotCache` 缓存过滤排序后的普通媒体列表；搜索输入使用短去抖，筛选、排序、排序方向、`libraryRevision` 或 `destination.id` 变化时应刷新快照，不要把全量 filter/sort 放回 SwiftUI `body`。重复选择同一排序项要切换 `LibrarySortOrder`，新排序项默认正序；排序菜单同时提供“评分”和“评级”，“评分”读取资料源 `rating`，“评级”读取用户 `userRating`，缺失值按 0 分处理。手动视频集合页额外提供“集合顺序”，默认保持 `VideoManualCollection.itemIDs` 的加入顺序；集合页右键排序调用 `VideoManualCollection.reorderedItemIDs` 只调整内部 ID 顺序，不触碰媒体文件；集合封面由 `VideoManualCollectionCoverView` 根据可见集合条目的前几张海报派生，侧栏预览通过 `videoManualCollectionPreviewItemsByCollectionID` 一次性批量构建，避免每行重复扫描可见媒体；其它普通分类不得显示这个排序项。快照缓存使用明确访问顺序做 LRU 淘汰，避免字典顺序导致缓存抖动；不要用 selection `.id` 强制重建来修复分类切换，否则会重新引入页面切换卡顿。目标快照未命中时用 `AppLoadingView` 轻量骨架占位，并把重算延后一轮主线程后写回，写回前必须校验当前 snapshot key，避免旧结果晚到覆盖新页面。
- `LibraryView` / `MusicLibraryView` 的快照重算必须保持后台化：MainActor 只允许读取当前输入快照、切换加载状态和写回最终结果，搜索匹配、watch filter、音乐筛选、排序、专辑/艺术家分组和 `MusicTrackRowModel` 生成应在 `Task.detached` 中完成。不要为了减少代码量把 `LibrarySnapshotBuilder` 或 `MusicLibrarySnapshotBuilder` 的逻辑合回 view body。
- 视频侧栏不再提供独立“最近播放”页，旧 `video-recent` 存储 ID 需要迁移到 `VideoLibrarySection.watching`。“正在观看”应从所有非音乐、非保险库条目中收集 `hasPlaybackTrace`，并只在该页显示清除播放记录入口。
- `LibraryView` 切换 `SidebarDestination` 时必须把新的 destination 显式传给刷新函数，并用 `visibleItemsDestinationID` 标记当前列表所属栏目；不要在无参 `onChange` 闭包里读取可能属于上一轮 view 的 `destination`，否则会出现标题变了但列表落后一拍、动漫/电视剧内容互换的问题。
- `MusicLibraryView` 切换歌曲、专辑、艺术家、歌单、最近播放等 `MusicLibrarySection` 时必须把新的 section 显式传给刷新函数，并用 `visibleContentSectionID` 标记当前快照所属分区；不要回到无参 `refreshVisibleContent()` 读取旧 section 的写法，否则会复现标题和内容不一致的问题。音乐快照缓存同样使用 LRU 淘汰，并把 `MusicSortOrder` 纳入 key；重复选择同一排序项切换正序/倒序，新排序项默认正序。歌曲、专辑、艺术家最多播放排序基于 `MediaItem.playCount`，专辑/艺术家按分组歌曲播放次数汇总；艺术家页筛选只保留 `.all`，排序只保留名称、作品数量和播放次数。切页时如果目标 section 已有快照就立即显示，否则显示 `AppLoadingView` 骨架并延后一轮主线程计算，不能继续显示上一分区内容伪装成当前页，也不能直接闪成“暂无”。
- `MusicLibraryView` 使用 `MusicTrackRowModel` 预计算歌曲行展示字段、歌词存在状态和时长文本；歌词存在性通过 `MusicLyricsPresenceCache` 后台暖缓存刷新，滚动路径和筛选快照中不要重新解析 URL、格式化时长或调用 `FileManager` 查歌词。
- 音乐歌曲列表可以保留厚玻璃容器、描边和柔和阴影，但不要在虚拟化 `List` 外再套会破坏复用的大面积裁剪、逐行重材质或全量滚动偏移监听；默认音乐封面行内直接绘制小图标，只让当前 hover 行启用轻量白色高光、描边和小幅放大，避免滚动时触发昂贵离屏合成或海报组件解码。
- 音乐歌曲列表、歌单歌曲明细、专辑页和艺术家页是 P0 性能路径，当前必须走原生虚拟化 `List`，不要恢复会随滚动逐批增长并保留历史行的超长 `LazyVStack`。列表仍保留厚玻璃容器、固定高度行、绘制层高光、左侧光带、封面微缩放、圆形元数据按钮和滚动期 hover 抑制；歌曲标题和文件名展示应去掉常见音频后缀。专辑页视觉仍为网格卡片，但由 `List` 行承载多个卡片列，让离屏专辑卡片可被系统回收。视频海报墙使用列表行承载的分块瀑布流，后续调整块大小或列宽时优先保持首屏足够、进入页面轻和滚动回收有效。
- 专辑页卡片封面应继续使用 `PosterImage(cacheTargetSize:)` 的稳定缩略图目标，不要让虚拟化行用实际卡片尺寸反复生成不同缓存桶，也不要为了“高清”把专辑页缩略图改回大尺寸解码。视频海报墙使用 `PosterGridList` 的分块瀑布流：块内按估算高度分配到最短列，块作为列表行虚拟化；不要恢复一次性超长 `LazyVGrid`。海报标签使用流式布局，优先同排，换行时让底部一行尽量排满。海报/专辑标题使用 `MarqueeText` 时，测量层不能参与父布局；长标题只能在固定卡片宽度内裁剪、渐隐和 hover 首尾衔接循环滚动，不能撑宽网格列。
- 视频海报墙 hover 放大只作用于封面绘制层，外层卡片、标题、标签、布局和命中区域保持稳定，并继续接入滚动期 hover 抑制与 Reduce Motion。用户可见说明文案保持简洁、平静、直接，避免内部实现术语和教程式表达。
- 详情页和设置弹层中可能出现几十行以上的搜索/匹配结果，也应使用原生虚拟化 `List` 承载并隐藏系统背景、分隔线，再把玻璃行卡片放入 list row。当前 TMDB/音乐元数据搜索结果、字幕搜索结果和音乐元数据候选歌曲列表都已走该路径；不要把这些弹层结果列表退回 `ScrollView + LazyVStack`，否则大量候选会在滚动后持续保留封面、按钮和玻璃行状态。
- 名称排序的纵向/信息流页面不再显示右侧首字母或首字符索引；如果后续重新设计定位提示，必须先验证视觉方案不会破坏当前玻璃列表观感，并且不能引入每帧滚动偏移监听或全局鼠标/滚动采样。
- 设置页主体使用原生 `List` 承载设置分组，每个分组作为独立行虚拟化并清除系统行背景；设置分组优先使用 `staticSurfaceBackground`，页面保持 `.glassPerformanceMode(.balanced)` 与 `.preferStaticGlassSurfaces(true)`，避免滑动时一次性布局所有区块或持续追踪 pointer 光效。不要把设置页退回 `ScrollView + LazyVStack` 或给分组恢复连续 pointer 光源。
- 不需要鼠标光源响应的长列表卡片和设置分组应优先使用 `staticSurfaceBackground`；它保留 `LiquidGlassSurfaceLayer` 的左上浅米白/香槟受光和静态边缘轮廓光，但走轻量渲染路径并不再开启连续 pointer 光源监听。不要给滚动热路径重新套 `surfaceBackground`。设置页主体使用 `.glassPerformanceMode(.balanced)` 和 `.preferStaticGlassSurfaces(true)`，设置首页标签这类网格按钮也不要恢复连续 pointer 边缘光。`glassFormField` 和 `GlassMenuButton` 属于普通页面重复控件，当前使用静态暖白玻璃填充、直接高光描边和轻接触阴影；不要在这些控件里重新加入 `.regularMaterial`、跨层 `blendMode` 高光或大投影。`GlassSearchField` 只在页头少量出现时通过 `HeaderControlGlassBackground` 使用更厚实、更明确的暖白系统玻璃，边缘折射和轻暗边用于拉开控件/背景层级，不能把这种实时采样扩散到长列表行。`ContentView` 在有音乐 `activePlayerItem` 时会给下层 `navigationRoot` 写入 `glassPerformanceMode(.balanced)`，保留普通页面的局部 pointer 光、按钮边缘光和封面检视，但降低强度、增大位移阈值并减轻倾斜幅度；只有音乐展开覆盖时才切到 `.minimal` 并停止下层命中测试。全局 pointer 光效半径、强度和倾斜幅度要保持当前克制档，不要再扩大成大范围泛光。底部迷你播放器自身也进入静态玻璃性能档，不再做鼠标驱动光源采样；它只保留专用透明玻璃表面、裁剪在底栏内部的静态专辑色柔光和克制轮廓染色，常驻进度条为轻量 SwiftUI 绘制控件，避免播放进度发布时热更新原生 slider 或整条底栏。
- 音乐展开页里的 `FloatingLyricsGlass` 同时服务歌词卡、控制栏、弹层和收起按钮；歌词卡、控制栏和收起按钮必须保持中性无色材质，新增前景玻璃面板时应复用同一套卡片级 pointer 采样，让内部按钮通过 `liquidPointerContext` 获得边缘光，不要再叠第二套 `pointerLiquidLight`、额外的 `onContinuousHover` 或专辑色浸染。播放队列弹层和底部迷你播放器是例外：它们为了滚动稳定性必须保持静态玻璃，不再接鼠标光源。
- 音乐展开页底板的专辑取色分两类权重：低频颜料场可以使用封面空间颜色形成底板深度，封面 glow / 玻璃受光只允许亮部作为光源。近黑/黑场必须经 `lightEmitterMask` 和 `lightEmitterWeight` 降权，只提供遮光和干净阴影；不要把黑场提亮成会发光的彩色底。`MusicPlayerMetalBackdropView` 的 `paintPalette` 当前使用约 38px 网格与 15px 柔化，shader 内再通过 `glassifyAlbumColor` 清洁暗部、低彩和灰棕/灰绿风险，并用 `harmonicShift` 对可发光彩色区域做连续和声偏移。后续调参应优先改这些集中函数和 `kGlowStrength` / `kWhiteVeilStrength` / `kChromaBoost`，不要把整窗背景改回 SwiftUI blur 或高饱和渐变。
- 滚轮和触控板滑动比拖动滚动条更容易触发 row hover、pointer 光效和检视倾斜高频重算；长列表 ScrollView 应接入 `suppressHoverEffectsDuringScroll()`，并让行级 hover/scale/tilt 读取 `suppressPointerHoverDuringScroll`。停止滚动后会恢复完整 hover 特效，不要用永久移除 hover 的方式换性能。
- 音乐播放进度发布给 UI 的频率当前低于 8fps，暂停态还会抑制微小时间抖动；不要为了歌词或滑杆把整个 `MpvPlayerController.currentTime` 恢复为高频根视图订阅。`MusicPlayerView` 根视图不能以 `@ObservedObject` / `@StateObject` 订阅播放器，`currentTime`、`lyricTime`、`audioSpectrumBands` 只能由进度条、歌词时钟、收起态进度环和频谱这些独立轻量节点订阅。轻量 observer 必须直接订阅具体 `@Published` 字段，不要再用 `controller.objectWillChange` 让音量、状态、进度、歌词互相触发。音乐展开页专辑背景光效不再使用本地播放时钟、`TimelineView`、音频采样或播放/暂停状态驱动；`AlbumGlassBackdrop` 和 `AlbumNearFieldIlluminationLayer` 不应观察 `MpvPlayerController`，只在专辑色、布局或挂载状态变化时重绘。动态背景光和收起态频谱不得用 SwiftUI `Canvas` 每帧重画：背景光走 `CAGradientLayer` 组合，频谱走自定义 `NSView` + `CAGradientLayer` 高度更新。歌词区域保留惰性/窗口化思路，当前只让 active 附近复杂渲染，非当前行保持 Equatable 静态文本；歌词换行时要优先做平滑的行级布局过渡，避免逐字放大、缩放和折返一起抢位置。不要未经 Instruments 证明就把歌词全量改成 `VStack`。展开首帧必须继续懒加载重型视觉：全屏高斯封面背景等 `glassLayerReady` 后再挂载，歌词卡片等 `entrancePhase` 至少进入第一阶段后再创建，暂停态封面不可见的模糊 glow 不应构建。底部迷你播放器的专辑光效必须裁剪在底栏内部。若后续需要更细的音频可视化，应单独做轻量采样通道，避免让整个 SwiftUI 播放器树跟着每帧重算。
- 首页、视频分类页、音乐分类页和 EMBY 页的扫描按钮应调用 `AppState.scanSources(for:)`，只扫描当前分类对应的媒体源；媒体源页的 `scanAllSources()` 保留为全量扫描入口。自动识别媒体源会跨类型导入，应继续通过媒体源页“扫描全部”触发。自动扫描复用同一队列，运行中不并发启动第二轮扫描。
- 扫描进度 UI 发布由 `ScanProgressThrottler` 控制，保留开始、结束、错误和关键进度变化，但不应每个文件都写 `scanProgress`。如果后续调整扫描器或新增进度字段，继续按时间/步进节流，避免大目录扫描时向 MainActor 投递成千上万次状态刷新。
- 数据库迁移中保留面向启动和列表的索引：`source_path`、`title COLLATE NOCASE`、`updated_at`、`last_played_at`、`type,parent_id`、`favorite`、`play_count`。批量清除播放记录和批量重置播放次数都使用 chunked `IN` 更新，不要恢复逐条 SQL 循环；扫描导入仍需后续评估事务化批量写入，但不要为了速度跳过现有稳定校验。
- `ArtworkImageCache` 使用 `NSCache` 保存本地海报、缺失路径和已知宽高比；本地海报通过 ImageIO 按目标显示尺寸下采样后再缓存，当前缓存总成本上限为 72MB。海报卡片 `body` 只允许读取已缓存比例、媒体分辨率兜底或默认比例；生成视频帧、默认封面和无正式海报路径的单视频应优先用 `MediaItem.resolution` 推导横版比例，不能为了排版同步加载 `NSImage(contentsOfFile:)`，也不要恢复原图级 `NSImage` 缓存。ImageIO 无法创建缩略图时应返回 nil 并显示占位，不要再用原图级 `NSImage(contentsOfFile:)` 兜底。
- 详情页文件存在性检查要走后台任务写回状态，不要在 `body` 中同步调用 `FileManager.default.fileExists`，尤其是 NAS 或文件提供器路径。
- 远程 http/https 海报和本地海报都走 `PosterImage` + `ArtworkImageCache`：远程图会下载后按目标尺寸下采样并进入同一缓存，避免 SwiftUI `AsyncImage` 在 hover、滚动或标签切换时重建并闪回默认海报。不要把 Emby 海报重新改回裸 `AsyncImage`。
- `ThumbnailGenerator.generateThumbnail` 先用 AVFoundation 截帧；失败后用 App 包内或系统路径的 `ffmpeg` 兜底，输出同一个媒体 ID 的 JPG，并继续做黑屏检测。不要把完整 mpv 可执行文件恢复为截帧工具。
- 清除播放记录只能更新播放位置、播放进度、已观看和最近播放时间，并同步更新 `AppState` 内存条目、当前队列和 `libraryRevision`；不要调用整库 `reload()`，否则音乐列表会短暂空白并让用户误以为需要重新扫描。

## UI 规则

- 页面卡片、选项栏和设置分组应保持偏白、干净、低噪声、液态玻璃风格；除音乐展开播放器外，普通页面需要保留从屏幕外左上方、靠近用户的位置斜向射入的宽范围低饱和浅米白/香槟环境光，`AppPageBackground`、`LiquidGlassSurfaceLayer`、按钮、搜索框、菜单和胶囊控件都应让左上受光面出现同方向偏白轻染色、高光和边缘轮廓光，右下只留轻暗边。不要把太阳光重新调成深蓝、高饱和泛光、水色泛光、反向染色或只照亮局部卡片的小光斑。通用 `SurfaceBackground` 要保留比侧栏更清晰的暖白填充、低噪声边缘和玻璃轮廓，设置页卡片也应复用同一层 `LiquidGlassSurfaceLayer`。普通页面中叠在玻璃卡片上的按钮、菜单、搜索框、输入框和胶囊控件需要通过暖白玻璃填充与下层玻璃拉开层级；只有侧栏、播放器暗色弹层、页头搜索/操作控件和少数非重复 hero 表面可以显式使用系统 material。`LiquidGlassButtonStyle` 的普通按钮路径应保持无 material、无 blur/screen 高光和轻投影，并裁剪到圆角避免直角底色；强调按钮、解锁按钮和获取类按钮要保持足够深的蓝色玻璃层与白色文字对比。首页标签、视频筛选和音乐筛选等同类胶囊控件统一复用 `GlassCapsuleControl`。
- 通用液态玻璃元素应保留鼠标光源响应：`pointerLiquidLight` 只在非重复、非滚动热路径的 hover 元素上根据本地鼠标位置更新高光，默认光效保持低饱和偏白，不要重新加重蓝色泛光，避免全局鼠标追踪导致长列表满屏重绘。连续 hover 更新必须经过 `PointerHoverThrottle` 节流，不要在卡片、按钮或检视倾斜上恢复每个鼠标事件都触发的 `@State` 写入。光效只能绘制在卡片背景层，按钮使用 `LiquidGlassButtonStyle` 或 `pointerLiquidEdge` 的边缘光，不能让发光层覆盖按钮内容。重复出现的列表行、海报卡、专辑/艺术家/歌单卡片、设置项及其 hover/selected 状态应按 `GlassSurfaceRole.repeated` / `repeatedSurfaceHover` / `RepeatedGlassButtonStyle` 留在 cheap 档，不要重新挂 material、大阴影或连续 pointer 采样；滚动结果行和媒体源行的小按钮也应优先使用 `RepeatedGlassButtonStyle`。视频海报和音乐专辑封面的检视倾斜使用 `pointerInspectTilt`，只作用在封面区域，不要让标题和卡片信息一起倾斜；视频/剧集海报卡片和音乐歌曲行可以保留轻微 hover 放大，但必须接入 `accessibilityReduceMotion` 降级。
- 全局动画统一使用 `AppMotion` 中的非线性 timing curve；新增动画不要直接使用短促 `.easeOut`，也不要给长列表和页面切换重新套大面积隐式动画。除音乐展开播放器外，页面材质应复用 `SurfaceBackground` / `AppPageBackground` 这套统一白玻璃语言。
- 左侧栏使用 `SidebarGlassBackground`，保持比正文卡片更透明、更克制的冷蓝液态玻璃质感，优先使用 `ultraThinMaterial` 和低蓝色洗色，蓝色洗色不要压过内容区。
- 首页 `HomeTabBar` 必须先尝试单行自然宽度，只有实际可用宽度放不下时才切到可横向拖动的双行布局；不要用固定数量阈值强制双行，也不要让右侧标签被窗口或工具栏遮住。
- 首页和首页设置只展示 `AppState.availableHomeTabs` 中有内容的入口；总览统计卡隐藏 0 项分类。
- 首页总览横向看板在 `mainLayoutTransitionActive` 期间必须保持被动：暂停自动滚动、横向拖拽监听和 hover/放大动画；离线入口与离线内容必须读取 `AppState.homeOfflineVideoItems` 派生缓存，不要在 SwiftUI `body` 中重新全库 filter 或逐条查询缓存状态。
- 首页总览统计卡片必须填满网格列并保持稳定高度，避免窗口宽度变化时内容贴左而卡片不对齐。总览横向卡片和海报墙视频卡片共用 `VideoItemContextMenuItems`，不要再在 `HomeView` 与 `PosterGridView` 分别维护菜单逻辑；本地条目保留重新分类，Emby/Jellyfin/Plex 条目不显示本地重分类入口。`nextUp` 看板的删除播放记录需要在单集本身无播放痕迹时提升到所属系列清理，确保下一集提示能随播放痕迹删除而消失。
- 任务中心重试只能基于 `BackgroundTaskSnapshot.retrySourceID` / `retryItemID` / `retryQualityID` 这类明确上下文重建任务，不得解析标题或详情猜测目标。完整扫描、远程同步和封面预热记录来源 ID；视频缓存、章节图和片头片尾检测记录媒体 ID，视频缓存额外记录清晰度 ID；元数据补充和一键清理由当前全局状态重建。增量扫描失败的重试应升级为对应来源的完整扫描。旧历史任务缺少上下文字段时不要显示重试按钮；同一来源或同一条目已有活跃任务时也不要允许重复重试。
- 应用内音乐符号和音乐默认封面必须保持蓝青主题，不要再引入紫色或粉色播放卡视觉。
- 筛选条不要再出现“范围/筛选/排序”等前置文字标签。
- 筛选条左右边距要一致，选项不贴边；普通视频库和音乐歌曲/专辑/艺术家页的筛选使用小胶囊玻璃按钮，不要改回系统 segmented picker。
- 设置页右侧控件应统一靠右对齐；长控件右边界和短按钮右边界保持一致，设置页里的 TextField/SecureField 输入文字居中显示，输入框宽度按当前文本长度在最小/最大宽度之间弹性变化。普通页面表单输入优先使用 `glassFormField`，让保险库、来源登录、元数据搜索和设置输入框共享暖白玻璃填充、焦点描边和边缘光。
- 普通弹窗和结果列表不要回到默认白色 `List` / 系统 segmented 控件外观；需要选择项时优先使用 `GlassCapsuleControl` 或静态 `LiquidGlassSurfaceLayer` 卡片，确保弹窗、搜索结果和主页面的光照方向一致。
- App 图标使用用户提供的方案 1 原图源；应用内图标使用统一蓝系无边框符号风格，不能加底板/边框，也不能所有图标套同一个模板。视频、音乐、保险库、媒体源、设置、元数据等必须有可一眼区分的主体图形。
- 设置页右侧菜单按钮优先使用 `GlassMenuButton` 这类固定宽度的自绘控件，避免系统 Picker 造成视觉边界不一致。
- 设置页内置播放器相关项要跟随播放器选择显隐，并按用途拆成“首页入口 / 外观与布局 / 视频播放与缓存 / 音乐播放 / 媒体库更新 / 元数据与匹配 / 字幕下载 / 封面与截图 / 账号与同步状态 / 数据、存储与诊断”等可扫描分组；新增设置入口应优先放到最具体分组，不要重新堆回一个大“播放”或“高级”段。视频窗口宽度、预览图模式和更多视频控制行为放在播放器齿轮菜单中；设置页只保留视频默认启动方式的安全回切入口、视频缓存位置/容量和音乐/库级设置。默认倍速/快进快退可在播放器齿轮菜单中调整，设置页保留视频播放与缓存里的通用控制入口；默认音量不要再作为设置项展示。
